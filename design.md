# Karatag 技術設計仕様書 (Design Document)

バージョン: V131.20

## 1. コア・アーキテクチャ

Karatagは、大局的な構造把握に強い **Whisper (ASR)** と、局所的な時間分解能に優れた **Wav2Vec2 (CTC)** を組み合わせた、二段階強制アライメントエンジンを採用しています。

### 1.1. 骨格（Skeleton）の確定 - Whisper
Whisperは楽曲全体の文脈を理解し、歌詞の「行」が音声のどの区間に存在するかを特定します。
- **バイアス付き推論**: 分割セグメントごとにそのブロックの歌詞を `initial_prompt` に与えることで、認識の確信度を向上させます。
- **ASR候補の抽出**: スコア、RMS（音量）、ピッチ（有声音）の3点から、信頼できるアンカーポイントを決定します。

### 1.2. 筋肉（Muscle）の補完 - CTC Segmentation
確定したアンカー間の空白を、Wav2Vec2によるCTC（Connectionist Temporal Classification）アライメントで埋めます。
- **強制アライメント**: 特定の区間に特定の文字列が「必ず存在する」という制約条件下で、確率が最大となる音素境界を1/100秒単位で探索します。

---

## 2. 物理的分割アライメント (Partitioned Alignment)

複雑な楽曲（繰り返しの多いサビ、長い間奏）でAIが迷子になる問題を解決するための核心機能です。

### 2.1. セグメンテーション理論
間奏タグ `[s - e]` を、単なる無音区間ではなく、楽曲を独立した複数の「演算ノード」に分割する **完全な境界（Boundary）** として扱います。
- **累積誤差の遮断**: 各セグメントは `--ignore` 相当のオフセットを伴って独立して解析されるため、前後のセグメントでの誤認（ワープ）が物理的に伝播しません。
- **時間軸のロック**: ユーザーが指定したタグ時間は「絶対的な真実」として固定され、AIの自由度を適切に制限します。

---

## 3. 三重バリデーション・システム

ハルシネーション（Whisperの嘘）を排除するための多層フィルタリング。

### 3.1. 順序制約 (Monotonicity Gating)
- **数理**: $T_{start}(i) \ge T_{end}(i-1)$
- 全てのアンカー候補は、時間軸において単調増加しなければなりません。最後のリフレインを曲の序盤に吸着させるようなワープを構造的に防ぎます。

### 3.2. 歌唱密度チェック (Phoneme Density Validation)
- **数理**: $Density = \frac{Length(Kana)}{Duration(s)}$
- 正常な歌唱は概ね $2.0 \sim 8.0 \, char/s$ の範囲に収まります。Whisperが提案した区間が $0.3 \, char/s$ 未満（極端な間延び）の場合、ノイズを歌と誤認したと判断して却下します。

### 3.3. 音響バリデーション (Acoustic Grounding)
- **RMS境界**: 区間の平均エネルギーが全体の閾値を下回る場合、無音と判断。
- **ピッチ存在フラグ**: Yinアルゴリズムによる基本周波数の抽出結果が皆無な区間は、非歌唱区間として却下。

## 4. 構造的パースと音素分配

`[表示|よみ]` 記法を正確に反映するためのアルゴリズム。

### 4.1. 等分展開アルゴリズム
表示文字数 $D$ と読みの音素数 $K$ に対し、各表示文字 $d_i$ に割り当てられる音素長 $k_i$ は以下のように算出されます。
1. $base = \lfloor K / D \rfloor$
2. $extra = K \pmod D$
3. $k_i = base + (1 \text{ if } i < extra \text{ else } 0)$
これにより、漢字一文字に対して複数のひらがな時間を正確に紐付け、KRAファイルとの1対1の比較を可能にしています。

## 5. インテリジェント自己診断レポート

正解データがない状況下での品質保証（QA）ロジック。

### 5.1. 統計的アノマリー検知
- **偏差解析**: 楽曲全体の平均密度 $\mu$ に対し、行ごとの密度 $d$ が $d > 3\mu$ または $d < 0.2\mu$ となる個所を「異常」としてフラグを立てます。
- **制約飽和検知**: 1文字の長さが `max_char_duration` (1.2s) に到達している率を算出し、時間軸の圧縮・伸長を検知します。

### 5.2. パラメータ推論
診断結果に基づき、以下の論理で対策を提案します。
- **高密度検出** $\rightarrow$ 間奏タグの不足 / `search_margin` の過小。
- **低密度検出** $\rightarrow$ `anchor_score` の過大 / ハルシネーション。
- **物理飽和検出** $\rightarrow$ `max_char_duration` 設定の不備。

---

## 6. 技術的制約と今後の課題
- **MIDIエンコード**: 日本語歌詞（Shift-JIS/UTF-8）をMIDIメタデータに埋め込む際のエンコード互換性の制限。
- **非線形配分**: ピッチ勾配を用いた「タメ・走り」のより精密な再現。
