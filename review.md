# Karatag タイムタグ精度レビュー

バージョン: V131.20  
レビュー日: 2026-06-14

---

## 0. 前提条件（本レビューの土台）

本レビューは以下の運用前提に基づく。これにより一部の指摘の根拠と優先度が変わる。

### 前提1: 入力は歌声のみ（伴奏分離済み）
ボーカル抽出済みの音声のみが与えられる。**伴奏・ドラム・楽器音は存在しない**。

この前提が与える影響:
- **RMS包絡線 ≒ 歌唱アクティビティ** に直結する。よって RMS によるオンセット検出・無音判定・エネルギー重心補間の**信頼性が大幅に上がり**、関連改善（問題1-2 / 問題7 / 設計Phase 2）の優先度を**引き上げてよい**。
- ピッチ（f0）抽出も伴奏由来の倍音に汚染されないため、有声/無声判定が明確になり母音オンセット特定に使える。
- 一方で「伴奏を歌と誤認する」誤りは原理的に発生しない。`auto_interlude_threshold` 等の**伴奏除去目的のバリデーションはほぼ不要**になる（無音＝間奏の検出には引き続き有用）。
- 新たな注意点として、分離アルゴリズム由来の**残響テール・にじみ（bleed）・ミュージカルノイズ**が無音区間に残り、これを歌と誤検知しうる。バリデーションの主眼は「伴奏除去」から「分離アーティファクト除去」へ移る。

### 前提2: 通常実行では正解データは存在しない（正解はデバッグ専用）
`.kra` 等の正解データは**デバッグ・検証時にのみ**与えられる。通常実行では存在を仮定してはならない。

この前提が与える影響:
- 実行時の唯一の品質保証は [evaluator.py](src/evaluator.py) の**統計的自己診断**（`_self_diagnose`）であり、その重要度が相対的に**最も高い**。
- パラメータ自動調整やフォールバック判断は、正解に依存せず**内部統計のみ**（被覆率・歌唱密度・単調性・RMS/ピッチ整合）で行わねばならない。
- 本書で「MAE 改善」と記す数値目標は、あくまで**デバッグ時（正解あり）に測定する検証指標**であり、実行時のロジックがそれに依存してはならない。
- [formatter.py](src/formatter.py) の KRA パース（問題10）は**デバッグ用ツールの一部**であり、本番経路の精度には影響しない。

## 1. アーキテクチャ概要

```
音声 → AudioProcessor (librosa, 16kHz mono)
           ↓
      LyricsAligner.run()
           ├─ 間奏タグあり → _run_partitioned() [aligner.py]
           │       ↓ Whisper で行アンカー抽出
           │       ↓ CTC segmentation で行境界確定
           │       ↓ _align_and_format() で文字均等分配
           └─ 間奏タグなし → aligner_legacy.LyricsAligner [aligner_legacy.py]
                   (ほぼ同じパイプライン)
           ↓
      KaraFormatter → [mm:ss:cc]文字 形式で出力
```

処理は「Whisperで大まかな行位置を決める → CTCで行境界を精緻化 → 残りは均等分配」の3段階。  
**精度劣化の主因は、最後の「均等分配」段階と、その前段階の誤差伝播にある。**

## 1.5. 実測検証に基づく重大発見（最優先）

`idol.kra`（正解）と `idol_final_check.result.txt`（本ツール出力）を実際に比較した結果、設計書・README に書かれた挙動と実装が食い違う**致命的な問題**が3点見つかった。

### 発見A: `alignment_mode` 設定が現行コードで完全に死んでいる（dead code）

`grep -rn "alignment_mode" src/` の結果、現行の [aligner.py](src/aligner.py) では `alignment_mode` は**読み込まれるだけで一度も参照されていない**。分岐ロジック（`if alignment_mode == "hybrid"` 等）は `src/aligner.py.bak` / `src/aligner.py.bak2` にのみ存在し、本番コードから消えている。

```
src/aligner.py:48:   self.alignment_mode = "linear"        # 代入のみ
src/aligner.py:73:   self.alignment_mode = config.get(...)  # 読込のみ
src/aligner.py.bak:320:  if self.settings.alignment_mode == "ctc":   # ← 旧版にしかない
src/aligner.py.bak:337:  elif self.settings.alignment_mode == "hybrid":
```

**影響:**
- [settings.ini](settings.ini) の `alignment_mode = hybrid`、プリセット `standard=hybrid` / `ballad=ctc` / `fast=linear` は**すべて無効**。実際には常に同一の「線形＋RMSスナップ」処理が走る。
- [README.md](README.md) の「`alignment_mode = ctc` を試してください」、[evaluator.py:70](src/evaluator.py#L70) の自己診断アドバイスは**ユーザーを誤誘導する虚偽の案内**になっている。
- ユーザーがどれだけプリセットやモードを変えても、文字タイミングの分配ロジックは変わらない。

**対策:** `alignment_mode` の分岐を本番コードに復活させる（`.bak` のロジックを移植）か、設定・README・evaluator から当該項目を削除して実態に合わせる。**精度改善以前に「設定が効かない」状態を解消することが最優先。**

### 発見B: 出力が常に「1文字1タグ」で、正解の「語/フレーズ単位グルーピング」と構造が異なる

正解 `idol.kra` の伸ばす旋律区間では、**複数文字が1つのタイムタグを共有**している（＝語頭でのみタグ付け）:

```
正解:   [00:00:54]無敵の[00:01:17]笑顔で[00:01:89]荒らす[00:02:26]メディア[00:02:73]
本ツール: [00:00:39]無[00:00:67]敵[00:00:95]の[00:01:17]笑[00:01:37]顔[00:01:57]で...
```

正解は「無敵の」の3文字を 0.54 にまとめているが、本ツールは 0.39 / 0.67 / 0.95 と1文字ずつ機械的に割り振っている。一方、早口区間（`今日何食べた?`）では正解も1文字ずつタグ付けしており、**正解データは「歌唱の塊（音節グループ）」単位で粒度を可変にしている**。

**影響:** 本ツールの均等分配は、語内部に存在しない架空の境界を打つため、カラオケ表示時に文字が不自然なタイミングで色付きする。

**対策:** [formatter.py](src/formatter.py) に「語/フレーズ単位グルーピング」モードを追加し、ルビ境界・形態素境界・無音間隔をもとに、同一音節グループ内は語頭時刻に丸める出力オプションを設ける（詳細は設計書参照）。

### 発見C: 行頭オンセットが一貫して約 0.15〜0.21s 早い（systematic early bias）

| 行 | 文字 | 正解 | 本ツール | 誤差 |
|----|------|------|----------|------|
| 1行目 | 無（語頭） | 0.54 | 0.39 | **-0.15s** |
| 6行目 | 今 | 17.13 | 16.92 | **-0.21s** |
| 9行目 | 何 | 22.99 | 23.15 | +0.16s |

行頭が系統的に早まる傾向がある。原因は複合的:
1. `search_margin`（1.5s）の後方展開でCTC探索窓が左に膨らむ
2. RMSデルタによるオンセットスナップ（[aligner.py:274-276](src/aligner.py#L274-L276)）が、子音立ち上がりやブレス音を拾って母音オンセットより前に吸着する
3. CTC が子音区間の先頭を境界として返す

**対策:** RMSスナップを「母音エネルギーのピーク到達点」基準に変更し、`search_margin` を非対称（後方を狭く）にする。詳細は設計書 Phase 2 参照。

## 2. 精度問題の根本原因と改善案

### 問題 1: 文字タイミングが完全均等分配（最重要）

**場所:** `aligner.py:277`, `aligner_legacy.py:191`

```python
e = s + (dur_total / max(1, n_chars))  # 全文字に同じ幅を割り当てている
```

**問題点:**  
行内の全文字が等しい時間幅を持つと仮定している。実際の歌唱では、母音が長く伸び、子音（特に促音「っ」や無声子音）は短い。ビブラートや節回しも無視される。

**改善案（優先度: 最高）:**

1. **Wav2Vec2の音素確率を文字単位に使う**  
   現在 `ctc_segmentation` は行境界しか返さない。`determine_utterance_segments` の代わりに、行内の各音素（かな1文字）ごとに `ctc_segmentation` を走らせることで、文字レベルの境界を取得できる。

   ```python
   # 現在: 行単位でCTC
   segments = determine_utterance_segments(params, utt_idx, char_probs, timings, gap_kana)
   
   # 改善: 1行をさらにかな文字単位でCTC → 各文字の start/end を取得
   # prepare_token_list に1文字ずつ渡し、各文字の timings を取得する
   ```

2. **RMSエンベロープによる音節開始検出を強化する**  
   現在の探索範囲 `±5 frames = ±0.1s` を、例えば CTC が確定した行境界の信頼度スコアで動的に広げる。

3. **促音・長音符・拗音を音素重みとして扱う**  
   ```python
   # かな1文字あたりの相対重みを設定
   MORA_WEIGHTS = {"っ": 0.5, "ー": 1.2, "ぁ": 0.6, "ぃ": 0.6, ...}
   # 均等分配の代わりに重み付き分配
   weight = MORA_WEIGHTS.get(kana_char, 1.0)
   duration = (total_dur * weight) / total_weight
   ```

### 問題 2: Whisper の word_timestamps を活用していない

**場所:** `aligner.py:188`, `aligner_legacy.py:111`

```python
w_segments, _ = self._whisper.transcribe(block_audio, beam_size=5, language="ja", vad_filter=True, initial_prompt=prompt)
```

**問題点:**  
`faster-whisper` は `word_timestamps=True` オプションで単語（形態素）レベルのタイムスタンプを返す。現在はセグメント（文単位）のタイムスタンプしか使っておらず、行アンカーの精度がセグメント粒度に留まっている。

**改善案（優先度: 高）:**

```python
w_segments, _ = self._whisper.transcribe(
    block_audio,
    beam_size=5,
    language="ja",
    vad_filter=True,
    initial_prompt=prompt,
    word_timestamps=True  # ← これを追加
)

# 各セグメントの word から細かいアンカーを取得
for seg in w_segments:
    for word in seg.words:  # word.start, word.end, word.word が取れる
        ...
```

これにより行アンカーの精度が大幅に改善し、CTC 前の初期整合性が上がる。

### 問題 3: アンカー探索ウィンドウの非対称問題

**場所:** `aligner.py:214`（パーティション版）vs `aligner_legacy.py:130`（レガシー版）

```python
# パーティション版: ウィンドウが狭い (5)
for si in range(w_ptr_b, min(w_ptr_b + 5, len(w_all_b))):

# レガシー版: ウィンドウが広い (20)
for si in range(w_ptr, min(w_ptr + 20, len(w_all))):
```

**問題点:**  
パーティション版はブロック内の Whisper 候補数が少ない前提で `5` に絞っているが、Whisper が細かく分割した場合は正しいアンカーを見逃す。特に間奏タグで分割した短いブロックでも問題になる。

**改善案（優先度: 中）:**  
`min(w_ptr_b + max(5, len(w_all_b) // len(lines) + 3), len(w_all_b))` のように、ブロック内候補数と行数の比率に応じてウィンドウを動的に決める。

### 問題 4: アンカースコアのデフォルト閾値が高すぎる

**場所:** `aligner.py:43`, `aligner_legacy.py:43`

```python
self.anchor_score = 0.9
```

**問題点:**  
（前提1により伴奏との混在は原因ではない。）歌唱はそもそも ASR にとって難物で、母音の引き伸ばし・メリスマ・ビブラート・ピッチ変動により Whisper の書き起こしが歌詞と完全一致しにくい。結果として `SequenceMatcher` スコアが 0.9（settings.ini では 0.98）を超えることが稀になり、アンカーがほとんど取れない。アンカーが全く取れない状態では CTC に渡す区間が「曲全体」になり、精度が大幅に低下する。

**改善案（優先度: 高）:**

- デフォルトを `0.75` 程度に下げ、代わりにバリデーション（RMS・ピッチ・密度）でフォールスポジティブを排除する。前提1により RMS/ピッチが歌声に直結するため、このバリデーションは高精度に効く。
- スコアを動的に調整する機構を導入（前提2に従い、判断は正解非依存の被覆率のみを用いる）:  
  「アンカーが全行の50%未満しか取れなかった場合、閾値を0.1下げて再試行」

### 問題 5: Wav2Vec2 への入力音声正規化が不安定

**場所:** `aligner.py:121`, `aligner_legacy.py:89`

```python
norm = audio / (np.max(np.abs(audio)) + 1e-6)
```

**問題点:**  
（前提1によりドラム等の打撃音は存在しないが）歌声ステム単体でもダイナミックレンジは広く、ベルティングの大声 vs ウィスパー/ファルセットの小声、さらに分離由来のスパイク状アーティファクトが最大振幅を支配すると、歌声本体の相対レベルが下がる。Wav2Vec2 は入力レベルに感度があるため、ログ確率が全体的に低下し CTC の信頼度が落ちる。

**改善案（優先度: 中→低）:** 前提1で打撃音の外れ値がない分、影響は限定的。ただしパーセンタイル正規化は依然として安全側で、低コストなので導入推奨。

```python
# ピーク正規化の代わりにパーセンタイルベース正規化
p99 = np.percentile(np.abs(audio), 99)
norm = np.clip(audio / (p99 + 1e-6), -1.0, 1.0)
```

### 問題 6: search_margin によるクロップ範囲の精度依存

**場所:** `aligner.py:242–257`

```python
ss, es = max(0.0, t1_e - self.settings.search_margin), min(len(audio)/SAMPLE_RATE, t2_s + self.settings.search_margin)
crop = audio[int(ss*SAMPLE_RATE):int(es*SAMPLE_RATE)]
...
line_boundaries[v] = (ss + seg[0], ss + seg[1])  # ss を加算してオフセット復元
```

**問題点:**  
- `ss` は float のまま `int(ss * SAMPLE_RATE)` でサンプル変換されるため、最大 `1/SAMPLE_RATE = 62.5 μs` の切り捨て誤差が発生する。  
- この誤差は後段の `ss + seg[0]` による絶対時間復元に蓄積される。  
- さらに `seg[0]` は CTC の `index_duration = 0.02s` の離散単位で返るため、最大 20ms の量子化誤差がある。

**改善案（優先度: 中）:**

```python
# ss を sample 単位で確定させ、浮動小数点誤差を排除
ss_samples = max(0, int(t1_e * SAMPLE_RATE) - int(self.settings.search_margin * SAMPLE_RATE))
ss = ss_samples / SAMPLE_RATE  # 実際のクロップ開始を sample 精度で保持
crop = audio[ss_samples:...]
```

### 問題 7: 線形補間が歌唱速度変化を無視する

**場所:** `aligner.py:286–301`, `aligner_legacy.py:200–215`

```python
s = t1 + (t2 - t1) * (bc / (tc or 1))  # 音素数に基づく線形補間
```

**問題点:**  
アンカーが取れなかった行を「隣接アンカー間を音素数で線形に分割」しているが、歌唱には「タメ（間を引き延ばす）」「走り（詰め込む）」が存在し、線形では近似が粗い。特にバラードや感情的な曲で誤差が大きくなる。

**改善案（優先度: 低）:**  
補間対象区間の RMS 包絡線の重心（エネルギー重心）を使って分配比率を補正する:

```python
# 補間区間の audio スライスで RMS の累積和から時間を推定
rms_seg = librosa.feature.rms(y=audio[int(t1*SR):int(t2*SR)], hop_length=320)[0]
cumrms = np.cumsum(rms_seg)
# 各行の音素比率を cumrms の正規化値に射影する
```

### 問題 8: VAD フィルタが柔らかい歌声を除去する可能性

**場所:** `aligner.py:188`, `aligner_legacy.py:111`

```python
w_segments, _ = self._whisper.transcribe(..., vad_filter=True, ...)
```

**問題点:**  
Whisper の内蔵 VAD（Silero VAD）はノイズや長い無音を除去するが、ピアニッシモ（pp）の歌声やファルセット、ウィスパー歌唱を無音と誤判定して切り捨てることがある。

**改善案（優先度: 低→中）:** 前提1により非歌唱区間は（分離アーティファクトを除き）ほぼ無音なので、Whisper内蔵VADに頼らず**自前の RMS+ピッチゲート**で区間を判定するほうが制御しやすく、柔らかい歌声の取りこぼしも防げる。

```python
# vad_filter を無効化し、カスタム音響バリデーション（RMS + ピッチ）で代替
w_segments, _ = self._whisper.transcribe(..., vad_filter=False, ...)
# その後の RMS / pitch_mask チェックで無声区間は除外済みなので重複排除
# 前提1: 残響テール/にじみを誤検知しないよう、ピッチ存在 AND RMS閾値の AND 条件で判定する
```

### 問題 9: 促音・長音符の音素カウントが過大になる

**場所:** `text_processor.py:36`, `text_processor_legacy.py:30`

```python
kana = re.sub(r"[^ぁ-んーa-zA-Z0-9]", "", kana)
```

**問題点:**  
`ー`（長音符）や `っ`（促音）は1モーラとしてカウントされるが、  
- `ー` は前の母音の延長であり CTC 上では独立したフレームを消費しない場合がある  
- `っ` は次の子音の閉鎖であり、実質的な持続時間が極めて短い  

これにより「ゆっくり → ゆっくり」を3モーラとして処理すると、「っ」に過剰な時間が割り当てられる。

**改善案（優先度: 中）:**  
音素重み辞書を導入し、均等分配の分母に使う `n_chars` をモーラ重み合計に置き換える:

```python
MORA_WEIGHT = {"っ": 0.3, "ー": 0.8}

def weighted_char_count(tokens):
    return sum(MORA_WEIGHT.get(t[0], 1.0) for t in tokens if t[1] != -1)
```

### 問題 10: 評価器（evaluator.py）の KRA パースが1文字ずつ展開時に開始時間を共有する

> **注（前提2）:** これは正解データを使う**デバッグ・検証経路のみ**の問題であり、通常実行のタイムタグ精度には影響しない。優先度は低いが、デバッグ時のMAE測定の正確性のために残す。

**場所:** `formatter.py:46–55`

```python
for i, char in enumerate(chars):
    line_data.append((char, t_start, t_start + 0.1))  # 同じ t_start
```

**問題点:**  
同一タイムタグ内に複数文字がある場合（例: `[00:10:50]きれいな`）、全文字が `t_start = 10.5` として扱われる。実際の正解データがどう作られているかによるが、MAE 計算時にこの近似が評価精度に影響する。

**改善案（優先度: 低）:**  
KRA フォーマットが1文字1タグの形式であれば問題ないが、複数文字を含むタグがある場合は終了タグから1文字ずつ時間を逆算する処理が必要。

## 3. 改善優先度まとめ

| 優先度 | 問題 | 期待される改善効果 |
|--------|------|------------------|
| 最高 | 文字タイミングの均等分配 | デバッグ測定で MAE 0.2s → 0.05s 程度まで削減できる可能性（※前提2: 実行時は自己診断で代替評価） |
| 最高 | `alignment_mode` の dead code 解消（発見A） | 設定・プリセットが実際に効くようになる（信頼性の前提） |

## 4. 追加的な観察事項

### 4.1 パーティション版とレガシー版の不整合
- `aligner.py`（パーティション）では `offset` を使って絶対時間を管理しているが、`last_time` の管理方法がレガシー版と異なる（`last_time = fixed[-1][2] - offset` vs `last_time = fixed[-1][2]`）。これが間奏を跨ぐ行の時間連続性に影響する可能性がある。

### 4.2 バリデーションログのタイポ
- `aligner.py:201`: `"ババリデーション却下"` → `"バリデーション却下"` （「バ」が重複）

### 4.3 --export-midi オプションが未実装
- `README.md` に記載があるが、`src/` 配下に MIDI 出力の実装が存在しない。

### 4.4 settings.ini が存在しない場合の動作
- `Settings.load()` は `settings.ini` がない場合 `return` して終わるため、ハードコードされたデフォルト値が使われる。デフォルト値（特に `anchor_score = 0.9`）がそのまま使われることに注意が必要。
